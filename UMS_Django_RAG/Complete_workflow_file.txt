1. PROJECT OVERVIEW
================================================================================

This project is a Django-based REST API backend for a UMS (University Management
System) chatbot. The goal is to build a chatbot that can answer student queries
about the university using RAG (Retrieval-Augmented Generation) techniques.

In this initial phase, we:
  - Set up the Django project from scratch.
  - Installed Django REST Framework for building APIs.
  - Created a basic chatbot app with a POST endpoint.
  - Tested the API to confirm everything is wired correctly.

This forms the foundation on which we will later integrate:
  - A knowledge base (PDF/text documents about the university).
  - An embedding model (to convert text into vectors).
  - A vector database (like FAISS or ChromaDB) for similarity search.
  - An LLM (like Google Gemini or OpenAI GPT) for generating answers.
  - Text-to-Speech (like Deepgram) for voice-based interaction.

9. TASK 8 — CREATE BASIC CHAT API VIEW
================================================================================

PURPOSE:
  The "view" is the heart of any Django endpoint. It's the Python function or
  class that receives an HTTP request and returns an HTTP response. In our case,
  we're creating a POST API that receives a chat query and returns a response.

WHAT WE DID:
  - Opened: chatbot/views.py

  - Replaced the default content with:

      ─────────────────────────────────────────────────────────
      from rest_framework.views import APIView
      from rest_framework.response import Response

      class ChatView(APIView):

          def post(self, request):
              query = request.data.get("query")
              return Response({"message": f"You said: {query}"})
      ─────────────────────────────────────────────────────────

CODE BREAKDOWN (LINE BY LINE):

  Line 1: from rest_framework.views import APIView
    - Imports APIView from Django REST Framework.
    - APIView is a class-based view specifically designed for building APIs.
    - It provides built-in features like:
        • Request parsing (JSON, form data, multipart)
        • Response rendering (JSON output)
        • Authentication & permission checks
        • Exception handling

  Line 2: from rest_framework.response import Response
    - Imports the Response class from DRF.
    - Response automatically converts Python dicts/lists to JSON format.
    - It also handles content negotiation (can return JSON, HTML, etc.)

  Line 4: class ChatView(APIView):
    - Creates a class called ChatView that inherits from APIView.
    - By inheriting APIView, our class gets all the API functionality.
    - The class name is descriptive: it's a View for the Chat feature.

  Line 6: def post(self, request):
    - Defines how to handle HTTP POST requests.
    - 'self' = reference to the class instance (standard Python OOP).
    - 'request' = the incoming HTTP request object, containing:
        • request.data = the parsed body (JSON, form data, etc.)
        • request.method = "POST"
        • request.headers = HTTP headers
        • request.user = the authenticated user (if any)

  Line 7: query = request.data.get("query")
    - Extracts the "query" field from the request body.
    - .get("query") safely returns None if "query" is not provided
      (unlike request.data["query"] which would throw an error).
    - Example: If body is {"query": "hello"}, then query = "hello"

  Line 8: return Response({"message": f"You said: {query}"})
    - Returns a JSON response with the echoed message.
    - f"You said: {query}" is a Python f-string (formatted string).
    - If query = "hello", response = {"message": "You said: hello"}
    - DRF's Response class automatically:
        • Sets Content-Type to application/json
        • Serializes the dict to JSON
        • Sets HTTP status code to 200 (OK)

WHY CLASS-BASED VIEW (NOT FUNCTION-BASED)?
  Django supports both styles:
    - Function-Based Views (FBVs): Simple functions decorated with @api_view
    - Class-Based Views (CBVs): Classes that inherit from APIView
  
  We chose CBVs because:
    - Each HTTP method (GET, POST, PUT, DELETE) gets its own method
    - Easier to extend and maintain as complexity grows
    - Better organization for the RAG chatbot (we'll add GET, error handling, etc.)

RESULT:
  ✅ ChatView class created with POST endpoint.
  ✅ Accepts JSON body with "query" field.
  ✅ Returns JSON response echoing the query.

10. TASK 9 — CREATE URL FILE FOR CHATBOT APP
================================================================================

PURPOSE:
  Django uses a URL routing system to map URLs to views. Each app should have
  its own urls.py file to keep URL definitions modular and organized.

  This file tells Django: "When someone visits /chat/, send the request to
  ChatView."

WHAT WE DID:
  - Created a NEW file: chatbot/urls.py
    (This file does NOT exist by default — Django doesn't auto-create it
    for apps. We must create it manually.)

  - Added the following content:

      ─────────────────────────────────────────────────────────
      from django.urls import path
      from .views import ChatView

      urlpatterns = [
          path('chat/', ChatView.as_view()),
      ]
      ─────────────────────────────────────────────────────────

CODE BREAKDOWN:

  Line 1: from django.urls import path
    - Imports Django's path() function for defining URL patterns.
    - path() takes a URL string and maps it to a view.

  Line 2: from .views import ChatView
    - Imports our ChatView from the same app's views.py.
    - The dot (.) means "from the current package" (relative import).

  Lines 4-6: urlpatterns = [...]
    - urlpatterns is a special variable name that Django looks for.
    - It must be a list of path() entries.

  Line 5: path('chat/', ChatView.as_view())
    - 'chat/'        → The URL path (will become /api/chat/ after project
                        URL inclusion — see Task 10)
    - ChatView.as_view() → Converts the class-based view into a callable
                           view function that Django can use.
    - .as_view() is REQUIRED for class-based views. It:
        • Creates an instance of the class
        • Routes the request to the correct method (get, post, put, etc.)
        • Returns the response

URL ROUTING FLOW:
  User sends request to: http://127.0.0.1:8000/api/chat/
    → Django checks project urls.py
    → Finds path('api/', include('chatbot.urls'))
    → Strips 'api/' and passes 'chat/' to chatbot/urls.py
    → Finds path('chat/', ChatView.as_view())
    → Calls ChatView.post() method
    → Returns the Response

RESULT:
  ✅ chatbot/urls.py created.
  ✅ /chat/ URL mapped to ChatView.

11. TASK 10 — CONNECT APP URLs TO PROJECT
================================================================================

PURPOSE:
  The app's urls.py is isolated — Django doesn't know about it yet. We need
  to connect it to the project's main urls.py using include(). This creates
  a hierarchy:

    Project urls.py → includes → App urls.py → routes to → Views

  This is like a tree structure:
    /                          (project root)
    └── api/                   (defined in project urls.py)
        └── chat/              (defined in chatbot/urls.py → ChatView)

WHAT WE DID:
  - Opened: ums_chatbot/ums_chatbot/urls.py

  - Modified it to:

      ─────────────────────────────────────────────────────────
      from django.urls import path, include

      urlpatterns = [
          path('api/', include('chatbot.urls')),
      ]
      ─────────────────────────────────────────────────────────

CODE BREAKDOWN:

  Line 1: from django.urls import path, include
    - path   → For defining URL patterns (same as before)
    - include → For including another URL configuration module

  Line 4: path('api/', include('chatbot.urls'))
    - 'api/'                   → All chatbot URLs will be prefixed with /api/
    - include('chatbot.urls')  → Tells Django to look at chatbot/urls.py
                                  for the rest of the URL matching.

  HOW THE FULL URL IS BUILT:
    Project prefix:  /api/      (from project urls.py)
    App path:        chat/      (from chatbot/urls.py)
    Full URL:        /api/chat/

NOTE:
  We removed the default admin URL (path('admin/', admin.site.urls)) for now
  to keep things simple. We can add it back later when we need the admin panel.

RESULT:
  ✅ Project urls.py updated.
  ✅ Chatbot app URLs included under /api/ prefix.
  ✅ Full endpoint: http://127.0.0.1:8000/api/chat/

REQUEST-RESPONSE FLOW (DETAILED):
  1. Client sends HTTP POST to http://127.0.0.1:8000/api/chat/
     Headers: Content-Type: application/json
     Body: {"query": "hello"}

  2. Django receives the request at manage.py's dev server.

  3. Django middleware processes the request:
     - SecurityMiddleware: Adds security headers
     - SessionMiddleware: Handles session cookies
     - CommonMiddleware: Normalizes URLs
     - CsrfViewMiddleware: CSRF protection (DRF exempts API views)
     - AuthenticationMiddleware: Sets request.user

  4. Django URL resolver matches the URL:
     /api/chat/ → project urls.py → chatbot urls.py → ChatView

  5. DRF's APIView processes the request:
     - Parses the JSON body into request.data
     - Identifies the HTTP method (POST)
     - Calls ChatView.post(request)

  6. ChatView.post() executes:
     - Extracts query = "hello" from request.data
     - Creates Response({"message": "You said: hello"})

  7. DRF renders the response as JSON.

  8. Django sends the HTTP response back to the client:
     Status: 200 OK
     Content-Type: application/json
     Body: {"message": "You said: hello"}

RESULT:
  ✅ API endpoint http://127.0.0.1:8000/api/chat/ is fully functional.
  ✅ POST request with JSON body works correctly.

14. KEY CONCEPTS EXPLAINED
================================================================================

  ┌─────────────────────────────────────────────────────────────────────┐
  │ CONCEPT: REST API                                                  │
  ├─────────────────────────────────────────────────────────────────────┤
  │ REST = Representational State Transfer                             │
  │ API  = Application Programming Interface                          │
  │                                                                    │
  │ It's a way for two programs to communicate over HTTP.              │
  │ Our chatbot API allows any client (web app, mobile app, Postman)   │
  │ to send a query and receive a response — all over HTTP using JSON. │
  │                                                                    │
  │ HTTP Methods:                                                      │
  │   GET    → Retrieve data                                           │
  │   POST   → Send/create data  ← (we use this for chat queries)     │
  │   PUT    → Update data                                             │
  │   DELETE → Remove data                                             │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ CONCEPT: Django MVT Pattern                                        │
  ├─────────────────────────────────────────────────────────────────────┤
  │ Django follows the Model-View-Template (MVT) pattern:              │
  │                                                                    │
  │   Model    → Defines data structure (models.py)                    │
  │   View     → Handles business logic (views.py)                     │
  │   Template → Renders HTML pages (templates/)                       │
  │                                                                    │
  │ For REST APIs, we skip Templates and return JSON instead.          │
  │ The View is the most important part — it processes requests.       │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ CONCEPT: Django Project vs App                                     │
  ├─────────────────────────────────────────────────────────────────────┤
  │ Project = The entire web application (ums_chatbot)                 │
  │           Contains settings, main URLs, deployment config.         │
  │                                                                    │
  │ App     = A specific feature module (chatbot)                      │
  │           Contains models, views, URLs for that feature.           │
  │                                                                    │
  │ A project can have MANY apps:                                      │
  │   - chatbot (handles chat queries)                                 │
  │   - accounts (handles user authentication)                         │
  │   - analytics (handles usage analytics)                            │
  │   etc.                                                             │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ CONCEPT: URL Routing in Django                                     │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                    │
  │ User Request: POST /api/chat/                                      │
  │        │                                                           │
  │        ▼                                                           │
  │ ums_chatbot/urls.py                                                │
  │   path('api/', include('chatbot.urls'))                            │
  │        │  strips 'api/' → remaining: 'chat/'                       │
  │        ▼                                                           │
  │ chatbot/urls.py                                                    │
  │   path('chat/', ChatView.as_view())                                │
  │        │  matches 'chat/' → calls ChatView                         │
  │        ▼                                                           │
  │ chatbot/views.py                                                   │
  │   ChatView.post(request) → Response({"message": "..."})           │
  │                                                                    │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ CONCEPT: Virtual Environment                                       │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                    │
  │ Without venv:                                                      │
  │   All Python packages install globally → version conflicts!        │
  │   Project A needs Django 4.2, Project B needs Django 6.0 → CRASH  │
  │                                                                    │
  │ With venv:                                                         │
  │   Each project has its own isolated Python + packages.             │
  │   Project A has Django 4.2 in its venv.                            │
  │   Project B has Django 6.0 in its venv.                            │
  │   No conflicts! ✅                                                 │
  │                                                                    │
  └─────────────────────────────────────────────────────────────────────┘
